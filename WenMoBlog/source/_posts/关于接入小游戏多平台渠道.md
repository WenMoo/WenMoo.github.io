#### 一、简介


目前小游戏往往会涉及多渠道的接入，来增加我们游戏的收入，这里作为开发，来杂七杂八记录这过程中遇到的问题



#### 二、接入前置


往往不同的平台有很多不同的必接能力，很多平台也有自己独有的功能，比如抖音的推荐直出，推荐复访，能够实现玩家在刷视频的过程中刷出游戏界面，直接进入游戏玩等等。



对于必接能力，一般而言会让运营去整理所需要接入的能力，以及他们需要的功能，这部分建议让运营列出需求checkList，排好优先级，然后让策划跟进需求的表现效果，出案子，我们按照优先级开发接入。



完成必接能力才能确保能在对应平台上审核通过。如果没有运营来处理这部分，那么我们就要根据平台的接入文档去接入了所需。



<font style="color:#DF2A3F;">第一次提审肯定会被各类问题打回，强烈要建议预足时间去处理，完成必接能力后，尽早提审先，也能根据反馈去及时修改。</font>



#### 三、开发注意问题


1、表格配置不同



有些时候我们面对不同的渠道的会有不同的策略，相同的一个功能会存在不同的数值，那么我们读取的配置就可能不同，所以读取配置的地方需要有意识的去做统一的接口去获取，那么我们就能从接口处去做处理区分。目前举个例子是 <font style="background-color:#FBDE28;">商店表 </font>的处理，因为体验卡在不同平台的价格是不一同的，而这个价格是我们根据表格去读取的，这里就要封装 <font style="background-color:#FBDE28;">商店表</font> 的获取，不这样做的话，如果多处做区别，非常容易导致下次修改中遗漏等情况的出现。

2、平台接入要做什么



我们接入一个渠道，基本包含的三大部分，<font style="color:#DF2A3F;">登录，支付，广告，</font> 分别也是各自平台的 Agent 实现，这些倒是没啥，参考微信的实现就好了，虽然比较繁琐，但是都有迹可循，具体细节就不展开了（参考灏哥的鸿蒙 / 淘宝接入有详细的介绍），再过来是<font style="color:#DF2A3F;">平台能力 </font>**的接入，比如也就是各类 API 的检查接入，虽然比如重启游戏的接口，游戏分享的接口，这部分就需要深度体验功能是否都完整可用。再过来的话就是需要处理 **<font style="color:#DF2A3F;">平台必接能力 </font>**，比如针对抖音就有 侧边栏复访 的必接能力等等，再过来处理我们的 **<font style="color:#DF2A3F;">埋点上报事件</font>**，我们目前一般是接入数数和引力，这里具体需要接入哪些根据运营所需。致辞应该差不多了，自测一遍后，找测试老师走一遍完整测试就差不多了~



> 注意平台上不同的系统可能会有不同的问题，在安卓上可行的不代表 IOS 也可行，测试过程中需要检验！其次，根据平台支持的能力，可以去完善对应的打包流程，比如目前微信，抖音，支付宝等都支持打包机打包，这里需要平台有对应的 cli 来执行命令行上传等，如果平台支持可以接入，大大滴优化打包体验~
>



单独说一说平台能力的处理，一般是建立统一的接口，然后对于每个渠道去实现所需的接口能力，因为往往我们业务开发的人和接入渠道的人都是分开的，所以对外通过接口暴露，对内实现接口功能。所以我们每个平台都有对应的类处理



举个例子，顶部的状态栏，因为在一些平台关闭按钮旁边还有别的功能按钮，会导致遮挡到我们原本的砖石栏的显示，最开始的问题是在抖音上遇到的，所以直接在业务中判断是抖音平台就缩放为 0.84，但后面不少平台也有类似的情况，那么这个就是针对不同平台的不同处理，就应该将逻辑放到对应平台中去获取。同理比如我们的分享，一些开关（游戏圈是否开启，复制功能等等）



![描述](image_04.png)



每一个渠道都用对应的类来做处理实现对应平台相关的能力，因为同样的一个功能，不同平台的具体实现方式有可能是不同的，以及一些配置等。如我们对于是否开启广告，我们只需要在对应平台中去设置开启关闭。归根结底，核心是希望对不同平台的东西和内容都收缩归纳到各自的地方，而不是各种散装，在业务中混乱的分布，这样无论我们是想要处理平台的差异，还是说对于业务中的修改都便捷安全。在业务中穿插各种平台的判断去做处理，绝对是灾难性的！我们无法保证这个东西，以后你还记得，或者换做别人之后还能清楚之前的。

#### 四、吐槽


在对接接入不同渠道中，不同渠道总有自己的想法和坑~这部分一方面建议在对应渠道的论坛中查询有没有相关内容，再者最有用的方式是直接沟通对面的技术。



#### 五、各类平台的问题
+ 抖音
    - 抖音的测试中，需要打开开发者工具，里面可以指定全平台入口打开指定的版本，这样比如我们再开发版点击侧边栏功能跳转，打开侧边栏后点击入口，就能够跳转的是当前开发版，而不是别的版本。



+ **华为**
    - 华为接入的时候其实有三大块部分，华为安卓，华为鸿蒙，华为快应用，接入这三块特别需要注意的是有对数据互通的要求！
    - 在接入华为快应用的过程中，最坑点是 **登录卡加载的问题**，是因为账号登录成功之后，登录前下载的资源无法访问。如果我们游戏进入前的一些预加载操作等，就会各种异常或者闪退，这里就得确保登录完成前没有下载。
    - 还有个问题是游戏游玩中会闪退，当时联系华为技术定位到的问题是因为开启了引擎分离（cocos creator 3.8.2），他们内部有部分功能冲突，后面解决方案是关闭引擎分离，具体的修复排查等他们处理了
    - 注意华为快应用的包，运行的环境还分为 花瓣轻游，以及 快应用加载器，需要注意，可能在两者上会有异同，需要在两者上都运行测试，保证没问题



+ **支付宝**
    - 没啥印象深的，印象中的一个是支付宝的支付接入中，有一个 <font style="color:rgb(51, 51, 51);">buyQuantity 的参数，需要符合支付宝中的价格档位</font>
    - 另外支付宝测试中，需要注意的是，我们可以开启联调的设置（右上角里面），不然测试，加桌 / 添首 等跳转到外部的功能没有跳转回来这个版本，会跳转到正式的
    - 注意，支付宝游戏圈的按钮创建不是同步的，不能类似微信那边创建紧跟着显示，打印可以看到一个ready 的参数为 false ，意思按钮的渲染或初始化状态未完全加载完成。这会按钮是没法点击跳转的游戏圈的，这里建议略微延迟一下即可。
    - 我滴龟龟，游戏圈参数中，有一个类型10的  “<font style="color:rgb(51, 51, 51);background-color:rgba(22, 118, 255, 0.06);">当天(自然日)用户在本圈子访问的时长</font>” 的，注意这里返回的不是时长，而是小于30s返回0，大于等于30s,小于60s返回1，大于等于60s返回2，时长的规则是这样的，2表示超过60秒 并且注意：这个是累计计算的，比如我第一次访问了1秒，第二次访问了29秒，第二次访问之后会返回1
